# Tool-Integrated Reasoning (TIR)

A small-scale replication of the [TORA paper](https://arxiv.org/abs/2309.17452) (ICLR 2024), comparing **Chain-of-Thought (CoT)** reasoning against **Tool-Integrated Reasoning (TIR)** for mathematical problem solving, including a simplified version of TORA's **Output Space Shaping (OSS)**. Where CoT models must perform all computation internally via natural language, TIR models interleave natural language rationale with executable Python code, offloading arithmetic and symbolic manipulation to a code interpreter. OSS further improves TIR by augmenting the training data with diverse correct trajectories generated via nucleus sampling.

## Goal

Replicate the core findings of the TORA paper -- that TIR outperforms rationale-only CoT, and that output space shaping further improves TIR -- using a deliberately small setup: **Qwen2.5-0.5B** fine-tuned on just 1,000 training examples from GSM8k. The aim is not to match TORA's absolute numbers (which used LLaMA-2 7B--70B and 16k GPT-4-generated examples), but to validate that these advantages hold even at minimal scale.

## Method

### Models

- **Base model:** [Qwen2.5-0.5B](https://huggingface.co/Qwen/Qwen2.5-0.5B) (494M parameters)
- **Data generation model:** [Qwen2.5-Math-7B-Instruct](https://huggingface.co/Qwen/Qwen2.5-Math-7B-Instruct) (already TIR-capable natively)

### Training Data

**CoT** -- 1,000 examples sampled from the GSM8k train split. Each example pairs a math problem with its ground-truth rationale-only solution (natural language chain of thought ending in `#### <answer>`).

**TIR** -- 1,000 examples from the same GSM8k problems, with trajectories generated by Qwen2.5-Math-7B-Instruct. Each trajectory interleaves natural language reasoning with Python code blocks and their execution outputs, following the TORA format:

````
<rationale>
```python
<code>
```
```output
<execution result>
```
<rationale with \\boxed{answer}>
````

Of the 1,000 generated trajectories, 963 produced correct answers and were used for training.

**TIR+OSS** -- The fine-tuned TIR model is used to generate additional candidate trajectories via nucleus sampling. For each of the 1,000 training problems, 5 candidate trajectories are sampled (temperature=0.8, top_p=0.95), with code blocks executed in a real sandboxed interpreter (not hallucinated). Only trajectories that produce the correct answer are kept, then merged with the original TIR dataset. This yielded 2,248 correct trajectories total (963 original + 1,285 new from OSS). The merged dataset is subsampled back to 1,000 examples for training, giving the model a more diverse set of correct solution strategies.

### Output Space Shaping (OSS)

TORA's output space shaping uses a larger teacher model to correct invalid trajectories. Our simplified approach skips the teacher correction and instead relies on diversity through sampling:

1. **Generate** -- The fine-tuned TIR checkpoint generates 5 candidate trajectories per training problem using nucleus sampling, with code blocks executed interactively in a sandboxed interpreter
2. **Filter** -- Each trajectory is graded against the ground-truth answer; only correct ones are kept
3. **Merge** -- Correct OSS trajectories are combined with the original TIR training data
4. **Retrain** -- A fresh Qwen2.5-0.5B is fine-tuned on the augmented dataset

This was run on [Modal](https://modal.com/) with OSS generation parallelized across ~20 GPU containers via `.map()` (batch size 50), reducing wall-clock time from ~74 hours (sequential) to ~4 hours.

### Training Setup

All three models were trained from the same Qwen2.5-0.5B checkpoint with identical hyperparameters:

| Parameter | Value |
|---|---|
| Epochs | 1 |
| Batch size | 2 |
| Gradient accumulation | 4 (effective batch 8) |
| Learning rate | 2e-5 (cosine decay) |
| Warmup | 3% of steps |
| Max sequence length | 2048 |

Loss is computed only on the response tokens (the question/prompt is masked).

### Evaluation

All three models were evaluated on 200 examples from the GSM8k test split using greedy decoding. The CoT model generates rationale-only solutions; the TIR and TIR+OSS models generate interleaved rationale + code, with code blocks executed in a sandboxed interpreter (up to 3 tool-use rounds, 5-second timeout per execution).

### Key Deviations from TORA

| Aspect | TORA | This Project |
|---|---|---|
| Base model | LLaMA-2 7B--70B | Qwen2.5-0.5B |
| Training data | 16k GPT-4-generated examples | 1,000 Qwen2.5-Math-7B-Instruct examples |
| Output space shaping | 64 samples/problem, teacher correction | 5 samples/problem, filter-only (no teacher) |
| Eval set size | Full test sets | 200 examples (GSM8k) |

## Results

### GSM8k (n_train=1000, epochs=1)

| Model | Correct | Total | Accuracy |
|---|---|---|---|
| CoT | 51 | 200 | **25.5%** |
| TIR | 75 | 200 | **37.5%** |
| TIR+OSS | 85 | 200 | **42.5%** |

| Comparison | Delta |
|---|---|
| TIR vs CoT | **+12.0%** |
| TIR+OSS vs CoT | **+17.0%** |
| TIR+OSS vs TIR | **+5.0%** |

TIR achieves a **47% relative improvement** over CoT. Adding OSS on top of TIR yields a further **13% relative improvement** (5.0% absolute), bringing the total improvement over CoT to **67% relative** (17.0% absolute).

### Example Outputs

**TIR** (correct) -- the model writes and executes code to compute the answer:

> *Problem: Marilyn's first record sold 10 times as many copies as Harald's. If they sold 88,000 copies combined, how many copies did Harald sell?*
>
> To solve this problem, we can set up an equation based on the information given. Let's denote the number of copies Harald sold as H. According to the problem, Marilyn sold 10 times as many copies as Harald, so Marilyn sold 10H copies. Together, they sold 88,000 copies, so we can write the equation:
>
> H + 10H = 88000
>
> This simplifies to:
>
> 11H = 88000
>
> Let's calculate this value using Python to ensure accuracy.
> ```python
> total_copies = 88000
> ratio = 10
> copies_harald = total_copies / (1 + ratio)
> print(copies_harald)
> ```
> ```
> 8000.0
> ```
> The number of copies Harald sold is **8000**.

**CoT** (correct) -- the model reasons entirely in natural language:

> *Problem: Tracy used a piece of wire 4 feet long to support tomato plants in the garden. The wire was cut into pieces 6 inches long. How many pieces did she obtain?*
>
> There are 12 inches in 1 foot, so 4 feet is 4\*12 = \<\<4\*12=48\>\>48 inches.
> The wire was cut into pieces 6 inches long, so Tracy obtained 48/6 = \<\<48/6=8\>\>8 pieces.
> `#### 8`

## Usage

### Run a full 3-way experiment (CoT vs TIR vs TIR+OSS)

```bash
python scripts/run_experiment.py \
    --dataset gsm8k \
    --n 1000 \
    --tir_data_path data/tir_gsm8k_1000.jsonl \
    --output_dir results/exp_gsm8k_n1000_oss \
    --eval_limit 200 \
    --oss \
    --oss_num_samples 5 \
    --oss_temperature 0.8 \
    --oss_top_p 0.95
```

### Run on Modal (cloud GPU, with parallelized OSS generation)

```bash
modal run scripts/run_experiment_modal.py \
    --dataset gsm8k \
    --n 1000 \
    --eval-limit 200 \
    --oss \
    --oss-num-samples 5 \
    --oss-batch-size 50
```

### Run without OSS (2-way CoT vs TIR only)

```bash
python scripts/run_experiment.py \
    --dataset gsm8k \
    --n 1000 \
    --tir_data_path data/tir_gsm8k_1000.jsonl \
    --output_dir results/exp_gsm8k_n1000 \
    --eval_limit 200
```

### Generate TIR training data

```bash
python scripts/generate_local.py --dataset gsm8k --n 1000
```

### Train individual models

```bash
# CoT baseline
python scripts/train_cot.py --dataset gsm8k --epochs 1 --output_dir checkpoints/cot_gsm8k

# TIR
python scripts/train_tir.py --data_path data/tir_gsm8k_1000.jsonl --epochs 1 --output_dir checkpoints/tir_gsm8k
```

### Evaluate

```bash
# CoT
python scripts/evaluate.py --model checkpoints/cot_gsm8k/final --dataset gsm8k --prompt_format cot --answer_format native

# TIR
python scripts/evaluate.py --model checkpoints/tir_gsm8k/final --dataset gsm8k --prompt_format tir --answer_format boxed
```
